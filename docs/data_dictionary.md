# Data Dictionary — PaySim E-Commerce Fraud Detection

## 0. Overview

- Source: Kaggle `rupakroy/online-payments-fraud-detection-dataset`.

- Active raw file: `data/online-payments-fraud-detection-dataset/PS_20174392719_1491204439457_log.csv`.

- Full rows: 6,362,620. Fraud rows: 8,213. Fraud prevalence: approximately 0.129%.

- Module 1 default artifact is a stratified 15% sample that preserves the natural fraud rate (for fast iteration); the final submitted pipeline is built with `--full` on all 6,362,620 rows.

- Synthetic context is generated by `src/synth_context.py` with `SEED=42`.

## 1. Base PaySim Columns

| Column         | Type     | Unit     | Valid range                             | Description                                                                                                   |
|:---------------|:---------|:---------|:----------------------------------------|:--------------------------------------------------------------------------------------------------------------|
| step           | int      | hours    | 1..743                                  | Simulation time; one step is one hour.                                                                        |
| type           | category | -        | PAYMENT/TRANSFER/CASH_OUT/CASH_IN/DEBIT | Transaction type; real fraud occurs only in TRANSFER and CASH_OUT.                                            |
| amount         | float    | currency | >=0                                     | Transaction amount and proxy for fraud loss.                                                                  |
| nameOrig       | string   | -        | C########                               | Origin customer id from PaySim; almost always single-use in the real file.                                    |
| oldbalanceOrg  | float    | currency | >=0                                     | Origin balance before transaction.                                                                            |
| newbalanceOrig | float    | currency | >=0                                     | Origin balance after transaction.                                                                             |
| nameDest       | string   | -        | C... or M...                            | Destination account; repeated customer destinations are useful mule candidates for later feature engineering. |
| oldbalanceDest | float    | currency | >=0                                     | Destination balance before transaction.                                                                       |
| newbalanceDest | float    | currency | >=0                                     | Destination balance after transaction.                                                                        |
| isFraud        | int      | flag     | {0,1}                                   | Target label.                                                                                                 |
| isFlaggedFraud | int      | flag     | {0,1}                                   | PaySim's built-in rule flag for very large transfers.                                                         |

## 3. Generation Parameters

- `n_customers`: 200000

- `reveal_rate`: 0.55

- `legit_noise`: 0.04

- `account_age_days`: legit=(6.0, 0.9), fraud=(5.0, 1.0)

- `high_risk_country`: legit=0.06, fraud=0.28

- `is_disposable_email`: legit=0.04, fraud=0.22

- `is_new_device`: legit=0.15, fraud=0.5

- `shipping_billing_mismatch`: legit=0.08, fraud=0.42

- `num_failed_payment_attempts`: legit=0.3, fraud=1.4

- `ip_offset_deg`: legit=(0.2, 1.0), fraud=(1.4, 1.1)

## 4. Design Notes & Limitations

- Faker identity columns (`customer_name`, `email`, `billing_city`) are analyst-facing context and should not be treated as real customer PII.

- Real `nameOrig` is almost always single-use, so synthetic customer velocity is illustrative. Later feature engineering should prefer real repeated `nameDest` aggregation for mule-account signals.

- PaySim balance fields are highly predictive; a realistic modelling scenario should compare models with and without post-transaction balance-reconciliation features.

- Synthetic risk fields are intentionally overlapping by using `reveal_rate` and `legit_noise`; they should support realism, not create perfect target leakage.
